# 🧠 Чистый Интеллект

Локальный AI-оркестратор с обнуляемым контекстом и иерархической памятью.

> *«Модель не помнит — она навигирует по координатам, которые сама создала до обнуления контекста.»*

---

## Суть

Все LLM страдают от одной проблемы: **контекстное окно ограничено**. Сессия идёт — окно заполняется — ответы деградируют — всё встаёт.

**Чистый Интеллект** решает это иначе:
- Контекст **обнуляется** после каждого обмена
- Память живёт **вне модели** — в иерархической памяти L1/L2/L3
- Модель не «помнит» — она **навигирует по координатам**
- Две модели работают в тандеме: **7B пишет память, 3B читает**
- Буфер всегда маленький → **стабильная скорость** независимо от длины сессии

---

## Ключевые концепции

### Логический буфер

Не «контекстное окно». **Динамическое рабочее пространство** (500–8000 токенов), размер которого зависит от железа пользователя и напрямую влияет на качество обработки логики.

| Железо | Размер буфера | Качество |
|--------|---------------|----------|
| RTX 4090 (24GB) | 8000 токенов | МАКСИМУМ |
| RTX 3060 (12GB) | 4000 токенов | ВЫСОКОЕ |
| CPU 16GB RAM | 2000 токенов | СРЕДНЕЕ |
| CPU 8GB RAM | 1000 токенов | БАЗОВОЕ |

**Цель:** уменьшить буфер до 2000 токенов без потери качества за счёт улучшения алгоритма дистилляции.

### Иерархическая память (L1 / L2 / L3)

Память работает как **фильтр по приоритету** — от общей логики до конкретных координат.

| Уровень | Что хранит | Формат | Приоритет |
|---------|-----------|--------|-----------|
| **L1** | Дистиллят логики — суть понятая из диалога, отфильтрованная от мусора, сжатая до нескольких токенов | JSON (~20 токенов) | 🔴 Высший |
| **L2** | Связи — граф связей между темами и сущностями, извлечённых из диалога | JSON/Graph (~30 токенов) | 🟡 Средний |
| **L3** | Координаты — полная карта проекта: где что лежит, к чему привязано | ChromaDB + NetworkX | 🟢 Базовый |

**L1** — это «что мы поняли». **L2** — это «как это связано». **L3** — это «где искать детали».

### Последняя координата

Перед обнулением контекста модель создаёт **«последнюю координату»** — GPS-маркер, который сохраняет:
- Тему обсуждения и статус
- Логическую цепочку (по шагам)
- Активные ветви и ожидаемые следующие шаги
- Ключевой инсайт одним предложением
- Координаты сущностей (файл:строка)

После обнуления модель смотрит на последнюю координату и **воскрешает логику** — как по GPS-маркеру находит весь маршрут.

### Двойная дистилляция

```
7B модель (Чат) — «профессор»:
  → Генерирует ответ пользователю
  → Создаёт координату (она «умнее» — точнее дистилляция)
  → ОБНУЛЯЕТ свой буфер

3B модель (Оркестратор) — «ассистент»:
  → Читает координату, созданную 7B
  → Находит релевантный контекст в L3
  → Формирует контекст для 7B
  → ОБНУЛЯЕТ свой буфер

Программа — «секретарь»:
  → Фильтрует (Intent, RAG, Graph) без LLM
  → Управляет бюджетом токенов
  → Обнуляет буферы
  → Хранит L1/L2/L3
```

### Три уровня чтения буфера

```
Уровень 0: ПРОГРАММА (Python, без LLM)
  → Фильтрует, сортирует, организует сырой буфер
  → Результат: подготовленный контекст

Уровень 1: МОДЕЛЬ ОРКЕСТРАТОРА (3B)
  → Читает подготовленный контекст
  → Навигирует по координатам
  → Формирует контекст для Чата

Уровень 2: МОДЕЛЬ ЧАТА (7B)
  → Получает только релевантный контекст
  → Генерирует ответ
  → Создаёт координату перед обнулением
```

### Три режима работы

| Режим | Когда | Модули | Дистилляция |
|-------|-------|--------|-------------|
| **CODE** | debug, generate, review, refactor | Intent + RAG + Graph + L1 + L2 + L3 | Строгая |
| **CHAT** | chat, explain, discuss, creative | Intent + L1 + L2 (без RAG/Graph) | Мягкая |
| **HYBRID** | mixed, recall, explain_with_code | Все модули | Сбалансированная |

### Настраиваемое обнуление

| Триггер | Описание |
|---------|----------|
| `token_threshold: 80` | Обнулить когда буфер заполнен на 80% |
| `topic_change: true` | Обнулить при смене темы (Intent detection) |
| `/reset` | Ручное обнуление пользователем |

## Установка

```bash
# Клонировать репозиторий
git clone https://github.com/Remchik64/pure-intellect.git
cd pure-intellect

# Создать виртуальное окружение
python3 -m venv venv
source venv/bin/activate

# Установить зависимости
pip install -e ".[dev]"

# Скачать модели
huggingface-cli download bartowski/Qwen2.5-3B-Instruct-GGUF --local-dir ./models
huggingface-cli download bartowski/Qwen2.5-Coder-7B-Instruct-GGUF --local-dir ./models
```

## Запуск

```bash
# Запустить сервер Оркестратора
python -m pure_intellect serve --port 8085
```

## Конфигурация

Переменные окружения (префикс `PI_`):

| Переменная | По умолчанию | Описание |
|------------|-------------|----------|
| `PI_HOST` | `0.0.0.0` | Хост сервера |
| `PI_PORT` | `8085` | Порт сервера |
| `PI_ORCHESTRATOR_MODEL` | `qwen2.5-3b` | Модель Оркестратора (3B) |
| `PI_CHAT_MODEL` | `qwen2.5-coder-7b` | Модель Чата (7B) |
| `PI_BUFFER_SIZE` | `auto` | Размер логического буфера (auto = по железу) |
| `PI_RESET_THRESHOLD` | `80` | Порог обнуления (% заполнения буфера) |
| `PI_RESET_ON_TOPIC_CHANGE` | `true` | Обнуление при смене темы |
| `PI_STORAGE_DIR` | `./storage` | Директория хранения |
