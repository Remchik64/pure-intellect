# 🧠 Contextor: История одного прорыва

*Книга-рассказ о том, как контекстное окно стало не ограничением, а буфером извлечения*

## Глава 1. Проблема, которую все видели, но никто не решал

Представь: ты работаешь с LLM. Разговор длится уже 20 минут. Модель начинает забывать, о чём вы говорили в начале. Контекстное окно переполняется. Что делает большинство?

— **«Нужно больше VRAM!»** — говорят одни.
— **«Купи модель с 128K контекстом!»** — предлагают другие.
— **«Просто начинай новый чат»** — советуют третьи.

Но проблема не в размере окна. Проблема в том, **как мы используем контекст**.

Contextor — это система, которая смотрит на контекстное окно не как на ограничение, а как на **буфер извлечения**. Как swap в операционной системе: не нужен 1TB RAM, если хорошо работает механизм подкачки.

## Глава 2. Архитектура: как всё работает

### 2.1. Контекст как буфер

Когда ты отправляешь сообщение в Contextor, происходит не просто «запрос → ответ». Происходит целый оркестровый концерт:

```
Пользователь → Буфер (8K) → Извлечение фактов → Мягкий сброс → Чистый буфер + Память
↑                                              |
└────────────────────────────────────────────────┘
```

Каждый цикл:
1. Буфер заполняется сообщениями (6-12 сообщений)
2. CCI оценивает когерентность — «разговор ещё в теме?»
3. При CCI < 0.55 → мягкий сброс
4. Координата сжимает ~50 сообщений в ~200 токенов
5. Факты сохраняются в иерархическую память
6. Буфер чистый, но модель «помнит» через координату и память

**8K контекста — это не «мало». Это оптимальный размер буфера для извлечения фактов.**

---

### 2.2. Иерархическая память

Память в Contextor — не просто список фактов. Это трёхуровневая система с жизненным циклом:

```
🔥 WorkingMemory (горячий буфер, ~2000 токенов)
    ↓ decay + eviction
💾 MemoryStorage (долгосрочное хранилище, безграничное)
    ↓ compression + archive
📦 Archive (мягкое удаление, восстановимо)
```

#### Fact — атом памяти

Каждый факт имеет жизненный цикл:

| Уровень | Что происходит | Размер |
|---|---|---|
| **RAW** | Оригинальный контент | 100% |
| **SUMMARIZED** | Сжатый LLM | ~30% |
| **ENTITY_ONLY** | Только имя/ссылка | ~10% |
| **ARCHIVED** | Удалён из active, но восстановим | — |

Факт имеет веса:
- **attention_weight** (0.0 — 1.0) — насколько факт важен сейчас
- **stability** (0.0 — 1.0) — насколько факт постоянный
- **is_anchor** — якорный факт, защищён от decay и eviction

#### WorkingMemory — всегда маленькая, всегда чистая

WorkingMemory — это как RAM в компьютере:
- Бюджет: 2000 токенов
- Горячие факты (attention_weight ≥ 0.6) — остаются
- Холодные факты (attention_weight < 0.15) — уходят в Storage
- Anchor facts — **никогда** не удаляются

```python
# Факт упомянут → touch() → вес растёт → остаётся в WorkingMemory
# Факт не упомянут → decay() → вес падает → уходит в Storage
# Anchor fact → не decay, не evict — всегда в памяти
```

#### MemoryStorage — долгосрочное хранилище

Storage — как жёсткий диск:
- Факты не удаляются безвозвратно — только сжимаются
- Semantic retrieval через embeddings (SentenceTransformer → Ollama → BM25)
- Горячие факты (reference_count ≥ 5) → promote обратно в WorkingMemory
- Холодные факты (attention_weight < 0.1) → компрессия

---

### 2.3. CCI — Context Coherence Index

Как система понимает, что разговор «сменил тему»? Через **CCI** — индекс когерентности контекста.

CCI работает так:
1. После каждого turn сохраняем CoherenceEntry с ключевыми словами
2. Для нового запроса считаем BM25 similarity с историей
3. Если score < 0.15 → сигнал «тема сменилась»
4. Недавние turns важнее (экспоненциальное затухание)

```python
# weight = e^(-0.3 * i) — недавние важнее
# score < 0.55 → soft_reset trigger
# score < 0.15 → needs_context_restore
```

CCI не просто таймер — это **адаптивный** механизм. Разговор может идти 30 минут без сброса (если тема одна), а может сброситься через 4 turns (если тема резко сменилась).

---

### 2.4. Мягкий сброс — не потеря, а сжатие

Когда CCI триггерит soft_reset, происходит не «забвение», а **архивирование**:

1. **Создаётся координата** — LLM (2B модель) сжимает всю историю в ~200 токенов:
   ```
   УЧАСТНИК: Ренат
   ПРОЕКТ: Contextor
   ТЕХНОЛОГИИ: Python, FastAPI, Ollama
   ЦЕЛЬ: AI-ассистент с контекстной памятью
   ТЕМА: Обсуждение контекстного окна и хардкода
   ```

2. **Координата сохраняется** как anchor fact (не decay, не evict)

3. **История обрезается** до последних 6 сообщений (3 turns)

4. **Факты остаются** в WorkingMemory

Результат: буфер чистый, но модель «помнит» через координату.

---

### 2.5. MetaCoordinator — управление ростом координат

Но есть проблема: каждый soft_reset создаёт координату (~200 токенов). После 20 сбросов = 4000 токенов только на координаты. Это уничтожает экономию.

Решение — **мета-координата**:

- Каждые 4 координаты → 2B модель создаёт одну мета-координату (~300 токенов)
- Старые координаты → архивируются на диск
- В prompt: **1 мета-координата + 1 последняя координата** (~600 токенов)

**Независимо от длины сессии — всегда стабильно ~600 токенов на координаты.**

```
Сессия 2 часа:
- Без мета: 20 координат × 200 токенов = 4000 токенов ❌
- С мета: 1 мета + 1 координата = ~600 токенов ✅
```

---

### 2.6. Dual Model Router — разделение ответственности

Contextor использует **две модели** с разными ролями:

| Модель | Роль | Температура | Max tokens |
|---|---|---|---|
| **Coordinator (qwen3.5:2b)** | Структура: координаты, теги, intent | 0.1-0.3 | 300-512 |
| **Generator (qwen3.5:9b)** | Качество: ответы пользователю | 0.7 | 2048 |

Зачем?
- 2B модель **быстрая** (1-2 сек) для структурированных задач (JSON, классификация)
- 9B модель **качественная** для развёрнутых ответов
- Если 9B недоступна → fallback на 2B

Это не «большая модель лучше». Это **каждой задаче — свой инструмент**.

---

### 2.7. ImportanceTagger — LLM-классификация фактов

После каждого ответа LLM, 2B модель анализирует диалог и классифицирует информацию:

| Категория | Что значит | Что делаем |
|---|---|---|
| **anchors** | Критически важные факты (имена, проекты, конфигурация) | → add_anchor() — не decay, не evict |
| **facts** | Важные факты для текущей сессии | → add_text() — обычный lifecycle |
| **transient** | Временная информация (вопросы, рассуждения) | → skip — не сохраняем |

Если LLM недоступна — fallback на rule-based extraction (регулярные выражения для имён, проектов, версий).

---

### 2.8. MemoryOptimizer — фоновая оптимизация

Каждые 5 turns запускается оптимизатор:

1. **Promote**: горячие факты из Storage → WorkingMemory (если reference_count ≥ 5)
2. **Compress**: холодные факты → повышаем compression_level (RAW → SUMMARIZED → ENTITY_ONLY)
3. **Archive**: очень старые неиспользуемые факты → мягкое удаление

```python
# Pressure > 0.8? → агрессивный evict (attention_weight < 0.2)
# Pressure < 0.8? → обычный decay
```

## Глава 3. Поток данных: от сообщения до ответа

Давайте проследим полный путь сообщения через Contextor:

```
Пользователь: «Ренат, давай обсудим контекстное окно по умолчанию»
```

### Шаг 1: Intent Detection
```
intent_detector.detect("давай обсудим контекстное окно")
→ IntentType.CHAT, confidence=0.8
```

### Шаг 2: CCI Evaluation
```
cci_tracker.evaluate("давай обсудим контекстное окно")
→ CoherenceResult(score=0.72, signal="coherent")
```
Тема связана с предыдущим контекстом — сброс не нужен.

### Шаг 3: RAG + Graph Search
```
retriever.search("контекстное окно") → []
graph_builder.search_nodes("контекст") → []
```
(Пока пусто — проектная память ещё формируется)

### Шаг 4: Context Assembly
```
_build_system_prompt():
  - Базовая инструкция: «Ты — Contextor, локальный AI-оркестратор»
  - Контекст из памяти: WorkingMemory.get_context(max_tokens=400)
  - Координаты сессии: MetaCoordinator.get_context_for_prompt()
  - Тип задачи: CHAT
```

### Шаг 5: Build Messages
```
messages = [
  {role: "system", content: system_prompt + memory_context + coordinates},
  {role: "user", content: "давай обсудим контекстное окно по умолчанию"},
  ...chat_history[:-1],
]
```

### Шаг 6: Generate
```
dual_model.generate(messages, temperature=0.7, max_tokens=2048)
→ Ответ от 9B модели
```

### Шаг 7: Memory Update
```
importance_tagger.tag(query, response):
  anchors: ["Ренат", "Contextor", "контекстное окно"]
  facts: ["num_ctx: 8192 hardcoded", "модели поддерживают 262K"]
  transient: ["давай обсудим"]

working_memory.add_anchor("Ренат")
working_memory.add_text("num_ctx: 8192 hardcoded")
working_memory.cleanup(turn=5, query=query, response=response)
```

### Шаг 8: Save
```
session.save(working_memory, storage, chat_history, turn=5)
```

## Глава 4. Мягкий сброс: момент истины

А теперь представь: разговор длится уже 15 turns. CCI падает.

```
CCI score: 0.42 < threshold 0.55
Should reset? Yes! Reason: low_coherence
```

Что происходит:

### 4.1. Создание координаты

2B модель анализирует всю историю и создаёт компактный снимок:

```
УЧАСТНИК: Ренат
ПРОЕКТ: Contextor
ТЕХНОЛОГИИ: Python, FastAPI, Ollama, ChromaDB
ЦЕЛЬ: AI-ассистент с контекстной памятью
ОБОРУДОВАНИЕ: Docker, qwen3.5 модели
ТЕМА: Обсуждение контекстного окна и хардкода num_ctx
```

~200 токенов вместо ~2000 токенов истории.

### 4.2. Сохранение в MetaCoordinator

Координата добавляется в активный список. Если их уже 4:

```
4 координаты → 2B создаёт мета-координату
Мета-координата (~300 токенов) объединяет ВСЮ историю
Старые 4 координаты → архив на диск
```

### 4.3. Обрезка истории

```
chat_history[-6:]  # Оставляем последние 3 turns
```

### 4.4. Результат

- Буфер: 6 сообщений вместо 30
- Память: все факты в WorkingMemory
- Координата: вся история сжата в ~200 токенов
- Мета-координата: все координаты сжаты в ~300 токенов

**Модель «помнит» ВСЁ, хотя буфер чистый.**

## Глава 5. Инновации Contextor

### 5.1. Контекст как буфер, а не ограничение

Это ключевая идеология. Большинство проектов LLM бьются над проблемой «контекстное окно мало → давайте расширим». Contextor подходит с другой стороны:

- Контекстное окно = **обнуляемый буфер для извлечения фактов**
- Пока буфер наполняется диалогом, система **извлекает суть**
- Когда буфер переполняется → **мягкий сброс** = сжатие и сохранение
- Буфер чистый, но модель «помнит» через иерархическую память

Это как swap в ОС: не нужен 1TB RAM, если хорошо работает механизм подкачки.

### 5.2. Адаптивный мягкий сброс

CCI — это не таймер. Это **адаптивный** механизм:

- Разговор может идти 30 минут без сброса (если тема одна)
- Может сброситься через 4 turns (если тема резко сменилась)
- Hard limit в 16 turns — защита от бесконечного роста

### 5.3. Мета-координаты

4 координаты → 1 мета-координата. Независимо от длины сессии, в prompt всегда ~600 токенов на координаты.

### 5.4. Dual Model

2B для структуры, 9B для качества. Каждой задаче — свой инструмент.

### 5.5. Anchor Facts

Якори — факты, которые **никогда** не удаляются и не затухают. Имя пользователя, название проекта, ключевые параметры — всё это anchor facts.

### 5.6. Semantic Retrieval с fallback цепочкой

SentenceTransformer (GPU, ~5ms) → Ollama embeddings (~200ms) → BM25 keyword matching.

Всегда работает, даже без GPU.

## Глава 8. Философия Contextor

Contextor — это не чат-бот. Это **система иерархической памяти для LLM**.

Традиционный подход:
- Большое контекстное окно = вся история в одном потоке
- Когда контекст переполняется → модель «забывает» начало
- Решение: расширять окно (дорого, ограничено VRAM)

Подход Contextor:
- **Контекстное окно = обнуляемый буфер**
- Пока буфер наполняется, система **извлекает суть**
- Когда буфер переполняется → **мягкий сброс** = сжатие + сохранение
- **Буфер чистый, но модель «помнит» через иерархическую память**

Это как swap в ОС: не нужен 1TB RAM, если хорошо работает механизм подкачки.

Или как конспект лекций: не нужно помнить каждое слово профессора, если у тебя хороший конспект.

Contextor делает этот конспект автоматически.
